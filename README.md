
# 결제 시스템 구축

회원 등급별 할인과 결제 수단별 중복 할인을 지원하는 결제 시스템입니다.
비즈니스 요구사항이 자주 바뀔 수 있다는 조건 하에, 유연한 확장과 데이터 불변성을
핵심 가치로 두고 설계했습니다.

---

## 1. 기술 스택

- **Language:** Java 21
- **Framework:** Spring Boot 3.4.4
- **DB:** H2 (In-memory)
- **ORM:** Spring Data JPA
- **Build Tool:** Gradle
- **Library:** Lombok
- **Test:** JUnit 5, Mockito, AssertJ

---

## 2. 실행 방법

```bash
# 애플리케이션 실행
./gradlew bootRun

# 테스트 실행
./gradlew test
```

---

## 3. 브랜치 구성

- **`main`**: 기본 요구사항 구현
- **`feature`**: 심화 요구사항 구현 (이력 관리, 중복 할인)

---

## 4. 핵심 설계 의도

### 4-1. 전략 패턴으로 할인 정책 추상화

요구사항의 "할인 정책은 수시로 추가/변경 가능"이라는 문장에 주목했습니다.
if-else나 switch로 구현할 경우 새로운 등급이 추가될 때마다 기존 서비스 로직을
수정해야 하며, 이는 OCP(개방-폐쇄 원칙)를 위반합니다.

이를 해결하기 위해 `DiscountPolicy` 인터페이스를 두고, 구현체를 **등급별이 아닌
계산 방식별**로 분리했습니다.

- `FixedDiscountPolicy`: 고정 금액 할인 (VIP)
- `RateDiscountPolicy`: 정률 할인 (VVIP)

등급과 정책을 분리함으로써 GOLD 등급이 추가되더라도 기존 정책 클래스를 재사용할
수 있으며, 새로운 방식의 할인이 필요할 때만 새 구현체를 추가하면 됩니다.

또한 등급과 정책을 매핑하는 책임은 `DiscountPolicyFactory`에 전담시켜, 새 등급이
추가되더라도 Factory만 수정하면 되고 `PaymentService`는 구체적인 정책을 전혀
몰라도 되는 구조로 설계했습니다. **각 객체가 자기 역할만 수행하도록 책임을
분리한 것**이 핵심 설계 의도입니다.

### 4-2. 할인 금액 방어 로직 (Policy 내부 처리)

VIP 고정 할인(1,000원)이 800원 주문에 적용될 경우 결제 금액이 음수가 되는
문제가 있었습니다. 이를 두 가지 관점에서 고민했습니다.

**1) 책임을 어디에 둘 것인가 — Policy vs Service**

방어 로직을 `PaymentService`에 두면 정책이 추가될 때마다 Service도 함께 수정해야
하는 결합이 생깁니다. 할인 금액을 계산하는 책임은 Policy에 있으므로, **"원가를
초과하지 않는 할인 금액을 보장하는 책임" 또한 Policy가 직접 지는 것이 자연스럽다**
고 판단했습니다. 이에 따라 `FixedDiscountPolicy`는
`Math.min(discountAmount, originalPrice)`로 자체 방어하고 있습니다.

**2) 예외 처리 vs 0원 처리**

등급 할인은 고객이 선택한 것이 아니라 **시스템이 자동 적용**하는 정책입니다.
이런 상황에서 "할인이 원가를 초과했다"는 이유로 결제를 실패시키면, 고객 입장에서는
예상치 못한 결제 실패를 경험하게 되어 사용자 경험이 악화됩니다.

따라서 결제 실패(예외)보다는 **최대한 혜택을 보장하되 음수 결제만 방지하는 0원
처리** 방식이 비즈니스 관점에서 더 적절하다고 판단했습니다.

### 4-3. 스냅샷 방식으로 할인 이력 관리

"정책 수정/삭제 후 과거 결제 데이터와 이력 보존" 요구사항을 만족시키기 위해,
`DiscountHistory`를 정책 테이블의 **참조(FK)가 아닌 값 복사(스냅샷)** 방식으로
설계했습니다.

만약 `DiscountHistory`가 `DiscountPolicy`를 FK로 참조했다면:
- **정책 수정 시**: 과거 결제 조회 시 변경된 값이 노출되어 데이터 정합성 훼손
- **정책 삭제 시**: FK 무결성 오류로 과거 이력 조회 자체가 불가능

이런 문제들이 발생합니다. 이를 방지하기 위해 결제 시점의 정책 정보
(`policyName`, `discountRate`, `discountAmount`)를 **원시 타입 값으로 복사하여
저장**하는 방식을 채택했습니다.

영수증에 비유하면, "오늘의 가격을 참조하라"가 아니라 **"4,500원"이라고 직접
인쇄하는 방식**입니다. 한 번 저장된 이력은 어떤 외부 변경에도 영향받지 않으며,
이는 `DiscountHistoryTest`에서 `@MockitoBean`으로 정책 변경을 시뮬레이션하여
검증했습니다.

## 5. main → feature 확장 및 개선 사항

`main` 브랜치의 기본 설계(전략 패턴 기반 할인 정책)는 그대로 유지하고, 심화
요구사항에 해당하는 기능만 추가했습니다. 이는 main 단계에서 이미 OCP를 고려하여
`DiscountPolicy`를 추상화해두었기 때문에, 중복 할인 요구사항이 추가되어도
기존 코드 수정 없이 새 구현체(`PointDiscountPolicy`)를 추가하는 것만으로
확장이 가능했습니다.


### 5-1. DiscountPolicy 인터페이스 확장 (개선)

`main`의 `DiscountPolicy` 인터페이스는 할인 금액 계산(`discount()`)만 책임졌지만,
`feature`에서 이력 관리가 추가되며 결제 시점의 **정책 메타정보(정책명, 할인율)**
도 필요해졌습니다.

이에 따라 인터페이스에 `getPolicyName()`, `getDiscountRate()` 메서드를 추가하여,
각 정책이 자신의 식별 정보를 스스로 제공하도록 책임을 확장했습니다. 이는 스냅샷
저장 시 `PaymentService`가 각 정책의 내부를 알 필요 없이 **인터페이스만 통해 필요한
정보를 얻을 수 있게** 하여, 캡슐화를 유지한 채 기능을 확장한 설계 결정입니다.

### 5-2. 이력 관리 (DiscountHistory) 추가

`main`에서는 `Payment`에 최종 금액만 저장했지만, `feature`에서는 어떤 정책으로
얼마가 할인되었는지를 `DiscountHistory`에 스냅샷으로 기록합니다.

`Payment` 하나에 여러 `DiscountHistory`가 연결될 수 있도록 `@ManyToOne` 관계로
설계한 이유는, 등급 할인과 결제수단 할인이 각각 **별개의 할인 이력**으로 기록되어야
정책 단위 분석이 가능하기 때문입니다.

### 5-3. 포인트 결제 시 중복 할인

`PointDiscountPolicy`(5% 정률)를 추가하여, 포인트로 결제할 경우 등급 할인이 적용된
금액에 5%를 추가로 할인합니다.

**할인 적용 순서**: 원가 → 등급 할인 → 결제수단 할인

등급 할인을 먼저 적용한 이유는, 결제수단 할인은 "최종 결제 방식에 대한 추가 혜택"
이라는 의미가 강하므로, **회원 등급 기반 할인이 먼저 적용된 금액에 붙는 것이
비즈니스적으로 자연스럽다**고 판단했기 때문입니다.

## 6. 테스트 전략

| 테스트 | 유형 | 검증 내용 |
|---|---|---|
| `FixedDiscountPolicyTest` | 순수 유닛 | 고정 할인 로직 + 원가 초과 방어 |
| `RateDiscountPolicyTest` | 순수 유닛 | 정률 할인 로직 + 0원 엣지케이스 |
| `DiscountOverlapTest` | 순수 유닛 | 등급 할인 + 포인트 중복 할인 금액 검증 |
| `DiscountHistoryTest` | 통합 (`@SpringBootTest`) | `@MockitoBean`으로 정책 변경 시뮬레이션 후 과거 이력 보존 검증 |

**전략**
- 도메인 로직(할인 계산)은 스프링 컨테이너 없이 **순수 유닛 테스트**로 빠르게 검증
- 데이터 정합성은 실제 DB 저장/조회가 필요하므로 **통합 테스트**로 검증
- 정책 변경 시나리오는 `@MockitoBean`으로 Factory를 교체하여 **실제 코드 수정 없이**
  시뮬레이션
